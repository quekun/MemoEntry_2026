1）创建了BuffBall类，依然通过多态实现Tank类增益，与WeaponBall不同的是，不需要创建不同的具体子类类如SpeedBoostBall，因为Buff和Tank自身相关，直接在Tank类里面添加对应函数实现即可，三种buff使用了6个函数，分别实现buff开始和buff结束，通过计时器把开始和结束函数联系一起



2）通过标签可以很方便地在基类获得子类的状态，在捡起一个球之后就为Tank类添加对应的标签，通过基类BasePawn生成子弹时判断子类是否有BounceUp也就是反弹增加的标签，来判断是否需要增加子弹反弹次数



3）在写ShieldBall护盾球时，我为了护盾的独立性，单独创建了一个ShieldCompoent，便于为任一actor自由选择是否加上护盾，这导致我遇到了较大问题

一是护盾的特效问题，即产生的无碰撞mesh不跟随actor而是在静止在关卡一特殊位置，花费了一整天的时间，最终通过输出调试日志的形式发现组件没有连接到根组件而是父挂件为null，尝试改变添加组件的时机，发现原因：我是在子类中创建的场景组件并且在子类中将它添加到根组件，而不是统一 Actor 借助构造函数统一管理组件树，那样组件Mesh不知道自己属于哪个 Actor，可能引发的权限问题： 只有 Actor 才有权通过 SetupAttachment 建立初始的物理/逻辑层级。如果在组件内部直接写 Mesh->SetupAttachment(this)，有时会因为当前组件（父级）尚未完成自身的附加（Attachment），导致连接失效。以后尽量在 Actor 层面通过 SetupAttachment(ParentComponent) 来构建完整的树状结构，而不是让组件去“自建”子树。

二是发现代码效率底下，我起初是通过在tick函数中判断Tank类是否有ShieldUp标签来设立护盾的隐藏或者可见，而且这使得该组件无法访问actor自身的数据，就无法对actor产生较复杂的作用，后来查阅资料后了解到：状态轮询（Polling）不是ue的宗旨，虚幻引擎的组件设计哲学是 事件驱动（Event-Driven）可以通过委托的方式，组件获取该actor指针并且挂到actor的组件上，在组件里面将actor的自定义委托绑定到组件内部的函数，这样就可以在actor里面不需要获取组件指针就可以访问组件函数，我通过自定义事件，绑定函数AddDynamic和广播BroadCast，实现了tank类和Shield类的事件驱动型交互
